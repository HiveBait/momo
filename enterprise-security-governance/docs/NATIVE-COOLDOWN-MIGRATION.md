# Migration Guide: Native Cooldown Implementation

## Overview

With GitHub's native `cooldown` feature, we can replace custom detection logic with first-class platform support. This guide documents the migration from custom security gate detection to native Dependabot cooldown.

## Architecture Before (Custom Detection)

```
Dependabot Creates PR → Security Gate Detects Dependabot → Applies Reduced Threshold → Blocks if Violated
```

**Issues:**
- ❌ PR already created (spam)
- ❌ Custom Python logic required
- ❌ Complex GitHub event parsing
- ❌ Developer sees failed checks

## Architecture After (Native Cooldown)

```
Dependabot Queries Package → Checks Publish Date → cooldown < threshold → ⏸️ SKIP PR Creation
```

**Benefits:**
- ✅ No PR spam
- ✅ Native GitHub feature
- ✅ No custom code needed
- ✅ Better developer experience

## Migration Steps

### Step 1: Deploy Native Dependabot Configuration

**Action:** Roll out `.github/dependabot.yml` with `cooldown` to all repositories.

```bash
# Deploy via bulk-lockfile-generator workflow
cd enterprise-security-governance

# Modify bulk-lockfile-generator.yml to include dependabot.yml deployment
# See implementation below
```

**Template to deploy:**

```yaml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "daily"
    cooldown:
      default-days: 7
      semver-patch-days: 3
```

### Step 2: Simplify Security Gate Logic

With native cooldown, the security gate becomes a **backup validation layer** instead of primary enforcement.

**Updated priority:**
1. **Primary:** Dependabot cooldown (prevents PR creation)
2. **Secondary:** Security gate (catches manual PRs and edge cases)

**Code changes:**

```python
# dependency_age_gate.py - Simplified logic
def validate_dependencies(workspace_path: str) -> List[PackageViolation]:
    """
    Validate dependency ages.
    
    Note: With native Dependabot cooldown, this primarily catches:
    - Manual dependency updates by developers
    - Edge cases where cooldown didn't apply
    - Repositories without dependabot.yml configured
    """
    violations = []
    allowlist = load_allowlist()
    cutoff_date = datetime.now(timezone.utc) - timedelta(days=AGE_THRESHOLD_DAYS)

    # Simplified - no Dependabot detection needed
    print(f"🔍 Scanning workspace: {workspace_path}")
    print(f"📅 Enforcing {AGE_THRESHOLD_DAYS}-day maturation policy\n")
    
    # ... rest of validation logic
```

### Step 3: Update Bulk Lockfile Generator

Extend the workflow to deploy `dependabot.yml` alongside lockfiles.

**Add to `.github/workflows/bulk-lockfile-generator.yml`:**

```yaml
      - name: Deploy Dependabot Configuration
        if: steps.detect.outputs.skip == 'false' && inputs.dry_run == false
        run: |
          # Copy template dependabot.yml
          mkdir -p .github
          cp ${{ github.workspace }}/security-tools/.github/dependabot.yml .github/dependabot.yml
          
          # Stage for commit
          git add .github/dependabot.yml
          
          echo "✅ Dependabot configuration deployed"
```

**Updated commit message:**

```

This PR adds:
- Dependency lockfiles for deterministic builds
- Dependabot configuration with 7-day cooldown policy

Changes:
- ✅ package-lock.json (npm dependencies)
- ✅ .github/dependabot.yml (supply chain protection)
```

## Rollback Plan

If native cooldown causes issues:

### Option 1: Disable cooldown per repo

```yaml
# Remove cooldown block from .github/dependabot.yml
# Fall back to security gate validation
```

### Option 2: Adjust thresholds

```yaml
cooldown:
  default-days: 3  # Reduce from 7 to 3
  semver-patch-days: 1  # Reduce from 3 to 1
```

### Option 3: Full rollback

```bash
# Remove all dependabot.yml files
gh api graphql -f query='...' | while read repo; do
  gh api -X DELETE /repos/HiveBait/$repo/contents/.github/dependabot.yml
done
```

## Key Decisions

### Decision 1: Keep Security Gate as Backup

**Rationale:**
- Catches manual PRs (developer-created)
- Handles edge cases (repos without dependabot.yml)
- Enforces break-glass allowlist policy
- Provides audit trail to Coralogix

**Verdict:** ✅ Keep both layers

### Decision 2: Remove Custom Dependabot Detection

**Rationale:**
- Native cooldown makes detection unnecessary
- Simplifies codebase
- Reduces maintenance burden

**Verdict:** ✅ Remove custom detection logic

### Decision 3: Standardize Cooldown Across Ecosystems

**Rationale:**
- Consistent security posture
- Easier to understand and audit
- Aligned with industry best practices (SLSA, NIST)

**Verdict:** ✅ Same cooldown for npm, pip, gomod
